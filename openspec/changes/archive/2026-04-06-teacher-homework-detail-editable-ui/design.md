## Context

当前 `HomeworkDetail` 页面是学生答题视角，核心能力集中在：题目切换、按题型渲染输入控件、保存草稿与提交。教师侧缺少可编辑作业详情视图，无法在同一布局下完成题目维护、发布时间管理与最终发布。

本次变更约束如下：
- 先完成前端交互与状态闭环，不接入真实接口。
- 保持与学生版一致的三栏布局认知（左工具栏、中间题目区、右题号索引）。
- 教师题号拖拽仅允许同题型内重排，不允许跨题型交换。

相关角色：教师（主使用者）、学生（需确保现有答题体验不回归）。

## Goals / Non-Goals

**Goals:**
- 在 `HomeworkDetail` 基础上提供教师可编辑模式，覆盖新增题目、编辑题目与参考答案、发布时间设置、发布锁定。
- 中间编辑区支持两类图片上传入口：题目图片、答案解析图片。
- 右侧题号区支持同题型拖拽排序，并同步更新当前激活题号。
- 发布动作替换提交语义，发布后二次进入页面保持只读并提示不可更改。

**Non-Goals:**
- 不接入后端接口，不定义服务端持久化协议。
- 不扩展教师批改、统计、导出等能力。
- 不支持跨题型拖拽。
- 不重构学生端题型渲染逻辑。

## Decisions

### Decision 1: 同页双模式，而非新增独立教师详情页
- 选择：保留 `src/pages/HomeworkDetail/index.tsx` 作为入口，根据角色或视图状态切换“学生答题模式/教师编辑模式”。
- 原因：最大化复用三栏布局与现有样式变量，减少重复 CSS 与路由扩散。
- 备选方案：新增 `TeacherHomeworkDetail` 独立页面。
- 不选原因：会复制大量布局代码，后续维护成本高，且不利于保持学生/教师视觉一致性。

### Decision 2: 在 HomeworkStore 内引入教师编辑态数据模型
- 选择：扩展 `HomeworkStore`，新增 `teacherDraft`（题目列表、开始/结束时间、发布状态、当前激活题）与编辑方法。
- 原因：现有页面已依赖 `HomeworkStore`，将教师编辑态放在同一 store 可避免跨 store 同步。
- 数据结构重点：
  - 题目基础字段：`id`、`type`、`title`、`score`、`options`。
  - 教师编辑字段：`referenceAnswer`、`analysis`、`questionImages`、`analysisImages`。
  - 排序字段：`orderInType`（同题型内顺序）与 `displayNo`（渲染序号，动态计算）。
  - 发布字段：`publishStatus`、`publishAt`、`startTime`、`endTime`。
- 备选方案：页面局部 `useState` 管理全部教师态。
- 不选原因：与学生态并存时状态源会分散，后续接接口时迁移成本高。

### Decision 3: 题号拖拽使用 `@dnd-kit/sortable`，按题型容器隔离
- 选择：右侧每个题型分组作为独立 `SortableContext`，只允许组内排序。
- 原因：`@dnd-kit` 对 React 生态和列表重排支持成熟，组内约束实现清晰。
- 备选方案：手写 HTML5 Drag API。
- 不选原因：可维护性差、边界处理复杂（拖拽反馈、键盘可访问性、触屏支持）。

### Decision 4: “添加题目”采用 Modal + Form 的分步输入
- 选择：左侧新增按钮触发弹窗，先选题型，再输入题目内容与参考答案（题型相关字段动态出现）。
- 原因：集中校验，能保证新增题目结构完整，避免直接在中间区域插入半成品。
- 备选方案：直接在中间区域内联新增。
- 不选原因：会打断当前题目编辑上下文，且内联校验难统一。

### Decision 5: 发布采用显式状态机并锁定可编辑控件
- 选择：状态机为 `draft -> published`，发布前弹窗确认，发布后禁用新增、编辑、拖拽、时间修改。
- 原因：符合“发布后不可更改”的业务约束，避免通过局部禁用遗漏入口。
- 备选方案：仅在发布按钮处阻断再次发布。
- 不选原因：无法阻止用户继续改动题目内容，语义不完整。

### Decision 6: 开始/结束时间使用 RangePicker + 前端校验
- 选择：使用 `DatePicker.RangePicker` 维护时间窗，校验 `endTime > startTime` 且发布前必须填写。
- 原因：统一输入体验并降低格式处理成本。
- 备选方案：两个独立 DatePicker。
- 不选原因：联动校验更复杂，容易出现起止反转。

## Risks / Trade-offs

- [Risk] 拖拽排序后序号、激活题、分组总分可能不同步 -> Mitigation: 排序操作统一走 store action，集中重算 `displayNo` 与分组统计。
- [Risk] 教师编辑字段增加后与学生答题字段混用导致渲染分支复杂 -> Mitigation: 将学生/教师渲染逻辑拆分为独立子组件，入口层只做模式分发。
- [Risk] 发布锁定遗漏某些交互入口（如图片上传、键盘输入）-> Mitigation: 通过 `isPublished` 在控件层统一透传 `disabled/readOnly`，并增加回归清单。
- [Risk] 无后端接口时数据仅内存存在，刷新丢失影响体验验证 -> Mitigation: 开发阶段允许使用 mock 初始数据，发布流程以交互正确性为验收标准。

## Migration Plan

1. 扩展 `HomeworkStore` 数据模型与 action，保留学生态兼容字段。
2. 改造 `HomeworkDetail` 页面为模式分发结构，新增教师工具栏与编辑区。
3. 引入并集成拖拽能力，先完成同题型内排序，再补齐样式态。
4. 增加发布时间窗与发布确认流程，打通“发布后锁定”。
5. 执行手工回归（学生模式、教师模式、发布前后状态切换）。

回滚策略：若教师模式出现高风险问题，可通过角色分支临时关闭教师编辑入口，保留学生模式原行为不变。
