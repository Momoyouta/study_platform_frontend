## Context

当前项目已有登录页与首页基础路由，但尚未形成统一的学习平台工作台布局，也缺少课程、资料广场、个人统计等业务入口的可视化承载。需求明确要求按给定示意图实现像素级一致的后台式页面结构，并限定左侧为单级菜单、首页与课程页先行落地、课程检索与新增仅保留交互占位。

技术约束为 React + React Router DOM + Less，且需遵循现有目录分层（pages/components/router/store）。因此本次设计重点是：在不引入新依赖的前提下，建立可扩展的页面壳层与路由组织，让后续接口联调和页面补齐可在当前结构上平滑推进。

## Goals / Non-Goals

**Goals:**
- 构建与设计图一致的学习平台主布局（顶部栏、左侧菜单、右侧内容区）。
- 提供四个一级菜单：首页、课程、资料广场、个人统计，并实现激活态联动。
- 实现首页与课程页两个可访问路由页面，课程页与首页共享结构但保持独立文件。
- 实现右上角用户头像区域悬浮菜单，含账号管理与退出登录操作入口。
- 将账号管理设计为右侧视图切换，不触发全页面跳转。
- 实现退出登录：清理 localStorage、重置 store，并重定向到登录页。
- 建立可扩展的布局层目录结构，支持未来新增多种 layout。
- 为“搜索课程”“添加课程”保留 UI 交互位，明确待接口实现状态。

**Non-Goals:**
- 不实现课程查询、课程新增的后端请求与数据持久化。
- 不交付资料广场与个人统计的完整业务页面内容，仅保留导航占位。
- 不实现退出登录后的后端会话失效接口联调。

## Antd Component Strategy

为提升一致性、可维护性与可访问性，本次界面优先采用 Ant Design 组件组合，并仅在视觉细节处使用 Less 补充样式。

组件映射约束：
- 主框架：使用 `Layout`（Header/Sider/Content）承载页面壳层。
- 左侧导航：使用 `Menu`（`mode=inline`），通过 `selectedKeys` 与路由状态联动。
- 用户区：使用 `Dropdown` + `Avatar` 组合，菜单项由 `menu.items` 配置。
- 首页/课程工具栏：使用 `Input.Search` 与 `Button`（primary + icon）承载搜索与新增入口。
- 课程列表：使用 `Card` 或 `List + Card` 组合展示课程条目。
- 空态占位：无课程数据时使用 `Empty`，避免自定义空态结构分散。

响应式约束：
- 使用 Antd Grid 或 `Grid.useBreakpoint` 驱动断点行为（而非仅依赖手写媒体查询）。
- 在窄屏下侧栏可折叠或改为抽屉式展示，保证首页、课程页首屏可用。

交互一致性：
- 菜单点击、下拉菜单点击等交互反馈优先复用 Antd 默认行为。
- 占位阶段消息提示统一使用 `message.info`，文案明确“接口暂未实现”。

## Decisions

### 1) 采用 Dashboard Layout 路由壳层承载主界面
- 决策: 新增一个工作台布局组件，顶部与左侧导航固定，右侧通过子路由渲染页面内容。
- 原因: 将视觉框架与业务页面解耦，避免首页/课程页面重复维护壳层代码。
- 备选方案: 在每个页面重复渲染顶部和侧栏。
- 不选原因: 重复代码高，后续新增页面时维护成本高且容易出现样式偏差。

### 2) 左侧菜单使用路由驱动的单级配置
- 决策: 用单一菜单配置数组定义 key、label、path，不引入二级树结构。
- 原因: 需求明确无二级菜单，配置化方式便于后续扩展资料广场/个人统计页面。
- 备选方案: 直接硬编码多个按钮并手写点击逻辑。
- 不选原因: 难以统一 active 规则和路由映射，扩展性差。

### 3) 账号管理采用“内容区切换”而非外部跳转
- 决策: 点击账号管理后切换到工作台内部路由视图（如 /account），仍保持同一布局。
- 原因: 需求要求不跳转独立页面，仅切换右侧路由内容。
- 备选方案: 弹窗式账号管理。
- 不选原因: 会改变信息架构，且不符合“路由页面切换”的要求描述。

### 4) 首页与课程页文件分离，共享可复用模块
- 决策: 保持 `src/pages/Home` 与 `src/pages/Course` 各自入口文件；可复用课程列表展示组件。
- 原因: 需求明确“课程页面暂与首页一致但要创建两个文件”，为后续差异化演进留接口。
- 备选方案: 单文件复用并以参数区分。
- 不选原因: 初期虽简洁，但会削弱目录语义，增加后续维护心智负担。

### 5) 布局抽象到 `src/layouts` 目录并通过子路由挂载
- 决策: 新建 `src/layouts` 目录，当前 Dashboard Layout 放置其中，业务页面通过 layout 下的 `Outlet` 渲染。
- 原因: 未来存在多 layout 可能（如管理后台 layout、简化内容 layout），提前建立目录边界可降低后续重构成本。
- 备选方案: 将布局代码写在 `App.jsx` 或页面目录中。
- 不选原因: 会导致布局与业务耦合，未来增加新 layout 时路由与组件职责不清。

### 6) 退出登录采用统一前端会话清理流程
- 决策: 在“退出登录”菜单项触发时，统一执行 localStorage 清理、MobX store 重置，并 `navigate('/login')`。
- 原因: 需求明确要求“清除localstorage、store，然后重定向到login页”。
- 备选方案: 仅清除 token 或仅跳转登录页。
- 不选原因: 会产生脏状态残留，可能出现重新进入后展示旧用户信息问题。

## Risks / Trade-offs

- [风险] 设计图无完整标注（间距/字号/交互细节）导致实现与预期有偏差。 → 缓解: 采用组件级样式常量（间距、圆角、颜色）集中管理，便于快速微调。
- [风险] 资料广场、个人统计暂无页面定义，先占位可能与后续需求冲突。 → 缓解: 将占位页面和菜单配置隔离，后续可无痛替换为真实页面。
- [权衡] 课程页初期与首页一致会存在重复结构。 → 缓解: 通过共享子组件降低重复，同时保留独立页面文件满足需求约束。
- [风险] localStorage 使用方较多，直接全量清理可能误删非鉴权数据。 → 缓解: 优先清理鉴权相关键，必要时封装白名单/黑名单清理策略。
- [风险] 后续引入多 layout 时路由层级复杂度上升。 → 缓解: 约定统一路由元信息（layoutKey）与目录命名规范，避免散落定义。

## Migration Plan

1. 新增或改造 Dashboard 布局与路由嵌套结构，确保默认落到首页。
2. 新增课程页面文件并接入菜单路由。
3. 补充账号管理视图占位路由，完成用户区下拉菜单交互。
4. 实现退出登录流程（清理 localStorage + store，跳转登录页）。
5. 将资料广场、个人统计配置为可导航占位路由（仅空态容器）。
6. 执行现有前端构建与本地手工验收（菜单切换、激活态、用户下拉、退出登录、页面可访问性）。

回滚策略:
- 若出现导航不可用或样式异常，可回滚至旧路由入口，仅保留原有首页路由。
- 新增页面与组件均为增量文件，回滚成本低。
