## Context

当前 Dashboard 中的账号管理路由已可访问，但页面仅有占位内容，无法承载真实资料维护。前端项目使用 React + Ant Design + Less，并且已有页面（如 Course）采用“入口页面 + 同级子组件”方式组织复杂内容。后端已提供头像临时上传、基础资料修改、手机号修改（含验证码发码/校验）、密码修改等接口，前端需要在现有请求层上补齐调用并建立明确交互流程。

## Goals / Non-Goals

**Goals:**
- 将账号管理页改造为三 Tab 结构（基本资料、修改头像、密码管理），每个 TabPanel 为独立组件。
- 基本资料中展示姓名、id、性别、手机号、学校名、角色，仅允许修改性别与手机号。
- 建立头像“双阶段提交”流程：先上传临时图，再以 `tempAvatarPath` 提交正式更新。
- 建立密码修改表单校验与接口提交流程，保证旧密码校验由后端负责，新密码一致性由前端预校验。
- 在现有 UI 风格下实现稳定布局与反馈，不追求像素级复刻。

**Non-Goals:**
- 不实现“注销账号”“语言”等其它 Tab。
- 不改动后端接口协议与短信验证码基础设施。
- 不引入新的全局状态管理方案或新的 UI 组件库。

## Decisions

1. 页面结构决策：账号管理主文件仅负责头部与 Tabs 容器，每个 TabPanel 拆分为同级组件文件。
- 备选方案 A：所有逻辑写入单一页面文件。
- 选型原因：单文件会导致状态与副作用耦合，后续扩展（注销账号、语言）可维护性差；拆分后可独立迭代并与现有 Course 目录风格一致。

2. 数据来源决策：进入页面时读取当前用户资料作为初始展示，编辑成功后局部刷新当前面板状态。
- 备选方案 A：每次切换 Tab 都重新请求资料。
- 选型原因：频繁请求会增加延迟与闪烁；使用一次加载 + 成功后同步更新可减少请求并保持数据新鲜度。

3. 手机号修改决策：前端提供“两步交互”能力，首次仅传 `newPhone` 触发验证码发送，二次携带 `code` 完成更新。
- 备选方案 A：强制一次提交（前端先拿验证码，再统一提交）。
- 选型原因：后端已通过 `code` 是否为空区分阶段，前端沿用该约定能最小化适配成本。

4. 头像修改决策：上传与保存拆分为两个动作，并在本地保存 `tempAvatarPath` 直到用户点击“保存头像”；头像展示 URL 统一通过 `VITE_FILE_BASE_URL` 拼接规则生成可访问地址。
- 备选方案 A：上传完成后立即提交 `updateAvatar`。
- 选型原因：即时提交会造成误上传难以撤销；显式“上传后保存”更符合用户预期并便于失败重试。

5. 密码修改决策：前端执行基础校验（非空、长度、确认密码一致）后再调用 `updatePassword`。
- 备选方案 A：全部校验交由后端。
- 选型原因：前端预校验能减少无效请求并提升反馈速度，同时不替代后端安全校验。

## Risks / Trade-offs

- [风险] 手机号验证码发送节流策略由后端控制，前端若无倒计时可能导致用户重复点击。 → [缓解] 前端在发码成功后展示剩余秒数并临时禁用再次发码按钮。
- [风险] 头像临时路径在用户离开页面后可能失效。 → [缓解] 仅在单次会话中持有 `tempAvatarPath`，失效时提示重新上传。
- [风险] 基础资料与 store 中用户信息可能短暂不一致。 → [缓解] 更新成功后同步刷新页面展示值，并在可用时触发用户资料重新拉取。
- [权衡] 不做像素级还原可加快交付，但视觉细节与原图存在差异。 → [缓解] 保证信息层级、交互位置和关键间距一致。

## Migration Plan

1. 新增账号管理三个 TabPanel 组件及样式文件，保持现有路由不变。
2. 扩展 `src/http/api.ts`：新增 `updateBasic`、`updatePhone`、`updateAvatar`、`updatePassword` 方法，并复用 `uploadImageTemp`。
3. 在 `AccountManage` 页接入新组件与接口，完成基本提交流程与错误反馈。
4. 本地联调后验证三条关键链路：修改性别/手机号、上传并保存头像、修改密码。
5. 回滚策略：若上线异常，可回退到本次改动前版本，恢复账号管理占位页。
